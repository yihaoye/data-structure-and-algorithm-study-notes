## 权衡配置灵活性与代码简洁性的取舍标准
代码来做还是配置来做的边界是什么 — "配置化" 经常被当成天然更好的选择，但实际上滥用配置是很多系统变得难维护的根源之一。几个比较可靠的判断依据如下。

### 核心判断标准: 变化的 "频率" 和 "决策者" 是谁
**该用配置的情况**:
- 这个值**会被非开发者频繁调整**（运维、SRE、产品、甚至客户自己），而且调整不需要理解代码逻辑本身 — 比如超时时间、重试次数、feature flag 开关、限流阈值
- 调整这个值**不需要重新走代码评审和发布流程**，而且这是刻意想要的效果（比如生产环境紧急降级，等不了走 CI/CD）
- 这个值在**不同环境/不同客户之间需要不一样**，但背后的逻辑是完全一致的（比如同一段处理逻辑，不同租户的限额不同）

**该用代码的情况**:
- 涉及**业务逻辑判断**、分支条件、多个值之间有依赖关系 — 这些一旦放进配置，配置本身就变成了一种 "隐藏的代码"，只是用 YAML 写的，可读性和可测试性反而更差
- 改动**需要配合代码逻辑同步修改**才有意义 — 如果配置 A 变了，代码里对应处理逻辑 B 也得跟着改，那这两者就不该分离，分离了会导致 "配置改了但代码没同步" 或者反过来这种不一致的 bug
- 需要**类型检查、单元测试**覆盖的地方 — 代码有编译期检查和 IDE 支持，配置文件通常没有（除非专门做了 schema 校验），出错更容易到运行时才发现

### "配置让开发更不方便" 常见的几种具体病灶
1. **过度参数化**：把简单的逻辑拆成一堆配置开关的排列组合，本来一个 if-else 就能表达清楚，结果变成配置文件里 5 个字段互相影响，开发者要在脑子里 "运行" 这个配置组合才能理解实际行为 — 这时候配置比代码更难懂，因为代码至少有语法结构和 IDE 跳转，配置只是数据，逻辑散落在读配置的那段代码里。
2. **配置膨胀成了 "用 YAML 写的编程语言"**：比如某些复杂的 CI/CD 配置、某些规则引擎的配置文件，写着写着开始出现条件判断、循环、变量引用 — 这时候其实是在用一个更差的语言（缺少调试工具、类型系统、测试框架）重新发明代码。
3. **改配置的反馈链路比改代码还长**：如果改一个配置需要走审批、需要重启服务、需要等配置中心同步、还不能本地方便地测试 — 那这个 "配置化" 带来的所谓灵活性其实是假的，反而比直接改代码发布更麻烦，因为代码至少有本地开发环境和单元测试能快速验证。

### 一个简单的自查问题
改动这个东西时，希望的是：
- "让不懂这块代码逻辑的人也能安全调整" -> 配置
- "这个改动需要理解上下文、可能牵扯其他逻辑、需要测试保证正确性" -> 代码

如果发现某个配置项，每次改动前都得去看代码才知道该填什么值、填错了会有什么后果 — 这说明它其实更该是代码，只是被错误地 "配置化" 了，配置在这里没有降低认知负担，反而增加了一层不透明的间接性。
