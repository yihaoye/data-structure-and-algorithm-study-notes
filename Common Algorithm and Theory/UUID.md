## UUID 算法
全称 Universally Unique Identifier  

UUID 按照标准方法生成时，在实际应用中具有唯一性，且不依赖中央机构的注册和分配。UUID 重复的概率接近零，可以忽略不计。  
规范的 `8-4-4-4-12` 格式字符串基于 UUID 的 16 个字节的记录布局：  
`xxxxxxxx-xxxx-Mxxx-Nxxx-xxxxxxxxxxxx` 四位数字 M 表示 UUID 版本，数字 N 的一至三个最高有效位表示 UUID 变体。在例子中，M 是 1 而且 N 是 a（10xx2），这意味着此 UUID 是变体 1、版本 1 UUID；即基于时间的 DCE/RFC 4122 UUID。  

Nil UUID 是一个特例，值为 00000000-0000-0000-0000-000000000000；也就是说，所有位都设置为 0。  

UUID 的实现思想本质上是：用统计学上足够大的随机空间替代中心化协调，通过牺牲存储空间和一定的有序性，换取分布式环境下无协调的唯一性生成能力。不同版本只是在 "唯一性来源"（时间、MAC、随机数、哈希）和 "额外属性"（如是否有序、是否可复现）之间做不同权衡。  

### v7 版本
较新，2024 年才标准化进 RFC 9562  
结构：48 位毫秒级 Unix 时间戳 + 74 位随机数
思想：兼顾 v1 的时间有序和 v4 的随机不可预测。因为高位（最左）是时间戳，生成的 UUID 天然按时间递增排序，解决了 v4 做数据库主键时索引随机写入、性能差的问题（*为什么放最高位能优化索引？B+ 树索引的本质是一棵按 key 大小排序的树。核心问题在于新插入的 key 落在树的什么位置。因为时间戳单调递增，新生成的 UUID 在数值上几乎总是比之前生成的大。这意味着新记录几乎总是插入到 B+ 树的最右端，行为跟自增整数主键如 AUTO_INCREMENT 非常接近 — 顺序追加，极少触发内部页分裂。剩下的随机位仍然会造成不连续，但影响被压缩到了极小的范围内，同一毫秒内生成的多个 UUID，时间戳部分完全相同，排序完全由后面的 rand_a + rand_b 决定 — 这部分确实是随机的，所以同一毫秒内的记录彼此之间顺序是乱的，但乱序范围被限制在同一毫秒这个极小窗口内，不会跨到几秒、几分钟之前插入过的数据页。因此实际效果是这一毫秒内的几条记录可能在同一个页或相邻页内部顺序打乱，但依然是在树的最右侧区域打转，不会跳回中间某个早已写满、已经刷盘的旧页，这个粒度的乱序对页分裂、缓存命中率的影响可以忽略不计*）  
  
| 字段 | 位数 | 说明 |
|---|---|---|
| `unix_ts_ms` | 48 位 | Unix 时间戳,毫秒精度,从 1970-01-01 开始 |
| `ver` | 4 位 | 固定值 `0111`(即十六进制 `7`),标识这是 UUIDv7 |
| `rand_a` | 12 位 | 随机数(也可选用作亚毫秒精度计数器) |
| `var` | 2 位 | 固定值 `10`,标识 RFC 4122 变体 |
| `rand_b` | 62 位 | 随机数 |
具体示例：
```
018f4d2e   -    1c3a   -   7f21   -   9c3e   -   8b1a2f3d4e5c
└───┬───┘       └─┬─┘      └┬┘        └┬┘        └────┬─────┘
时间戳前半   时间戳后半+ver  rand_a  var+rand_b的高位  rand_b剩余
```

UUID v7 的代码实现很简单，核心就是 "时间戳 + 版本位 + 高质量随机数 + 位拼接"；真正困难的是 RFC 在位布局、随机位数、排序特性、碰撞概率、兼容性等方面做出的设计与验证和数学考虑，即这些规范才是困难与价值所在。
